D-RoCM (Distributed Robot Control Middleware) 产品需求文档1. 产品概述与设计目标1.1 产品定位D-RoCM 是一款针对工业机器人边缘计算场景设计的高性能、可观测、异步化通讯中间件。它并非为了取代 ROS 2 等通用框架，而是作为一个轻量级替代方案，验证在内存受限、极低延迟要求下，利用现代 C++ 特性（C++20/gRPC）能达到的性能极限。1.2 核心目标（面试加分项）极致性能：通过 C++20 协程与异步 gRPC 极大化提升并发吞吐。低资源占用：消除通用框架的冗余依赖，实现极小的内存脚印（Memory Footprint）。工程透明度：内置详细的性能埋点与分析工具，用数据证明系统稳定性。2. 核心技术架构项目采用“中心化发现 + 去中心化通讯”的混合架构：Registry (Master)：负责节点注册、心跳维护与服务发现。Node (Peer)：实际的业务节点，支持 Unary、Streaming 等通讯模式。Core-Lib：基于 C++20 封装的异步通讯库，屏蔽 gRPC 原始 CompletionQueue 的复杂性。3. 详细功能列表3.1 异步通讯模块 (Communication Engine)C++20 Coroutine Wrapper：封装 gRPC 的异步调用，支持 co_await 风格的非阻塞 RPC 请求。多模式消息支持：控制流 (Sync/Async Unary)：用于高可靠性的指令下发。感知流 (Server Streaming)：用于激光雷达等高频传感器数据上报。闭环流 (Bidirectional Streaming)：用于毫秒级远程控制回路。3.2 节点生命周期管理 (Node Orchestration)心跳与健康检查：采用分布式心跳机制，Master 节点在秒级内感知 Peer 离线。自愈机制：支持指数退避（Exponential Backoff）的自动重连。3.3 性能观测系统 (Observability - 替代 UI)数据埋点 (Metrics)：实时统计 QPS、端到端延迟（Latency）、序列化/反序列化耗时。追踪 (Tracing)：记录 Request-ID 在分布式节点间的流转时间。导出器 (Exporter)：支持将性能数据导出为 JSON 或 Prometheus 格式，用于离线分析。4. 技术栈与硬核特性 (Tech Stack)维度技术选型核心价值（为什么选它）语言标准C++20利用 Coroutines 解决异步回调地狱，提升代码可读性与执行效率。协议层gRPC / Protobuf工业级标准，零拷贝思想，跨语言兼容。并发模型Boost.Asio /线程池实现自定义的 Executor，根据 CPU 核心数优化线程调度。数据结构无锁队列 (Lock-free)在高频心跳和消息路由中使用原子操作，消除锁竞争。构建/部署CMake + Docker确保在不同工业电脑/嵌入式设备上的环境一致性。5. 非功能性需求 (Performance Focus)5.1 性能指标 (Hard Targets)吞吐量：在双核 Linux 环境下，支持每秒解析处理 200,000+ 条小型 Protobuf 消息。延迟：局域网内单向消息 P99 延迟应稳定在 1ms 以内。抖动控制：连续运行 48 小时，延迟抖动（Jitter）范围不超过平均值的 10%。5.2 鲁棒性优雅降级：当 Master 节点崩溃时，现有节点间的持续 Streaming 通讯不应中断。零拷贝优化：在消息分发过程中，尽可能减少不必要的内存拷贝（std::move, string_view）。6. 功能优先级与交付路线 (Roadmap)6.1 Phase 1: 核心骨架 (MVP)完成基于 Protobuf 的 RobotStatus 与 RobotCommand 协议定义。实现基于 gRPC Async API 的基础 Node 通讯。交付物：一个简单的 Echo 测试程序，展示异步调用的连通性。6.2 Phase 2: 异步演进 (Hardcore)引入 C++20 协程，将 gRPC 的 CompletionQueue 封装为协程调度器。实现 Master 节点的自动发现逻辑。交付物：支持 50+ 虚拟节点稳定运行的中间件内核。6.3 Phase 3: 性能画像 (Observation)集成 ghz 或自定义压测脚本。编写详细的性能 Benchmark 文档，包含火焰图分析。交付物：一份足以打动面试官的 《D-RoCM 性能白皮书》。7. 给开发者的建议，专注于“为什么”：在 README 中详细解释你为什么选择 C++20 协程而不是简单的多线程，这体现了你的架构思维。代码品味：利用 AI 辅助生成大量的单元测试（GTest），确保核心逻辑的内存安全（RAII）和异常安全。压测为王：不需要华丽的 UI，但在项目目录下一定要有一个 scripts/benchmark.sh。当你在面试现场能随手跑出几万 QPS 并在终端输出完美的延迟曲线图时，你的价值就无法被忽视。
